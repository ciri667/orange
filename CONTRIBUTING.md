# 贡献指南

感谢你关注橘记（Orange）。我们欢迎缺陷报告、功能建议、文档改进和代码贡献。

## 行为准则

请在讨论中保持友善、尊重和建设性。不要发布骚扰性内容，也不要在 Issue、Pull Request、日志或截图中泄露 API Key、访问令牌、笔记正文、个人身份信息或其他敏感数据。

## 开始之前

- 请先搜索已有的 Issue 和 Pull Request，避免重复工作。
- 较大的功能、涉及数据存储、文件写入、模型调用或 IM 集成的变更，请先创建 Issue 说明目标和方案。
- 本项目当前主要构建 macOS 桌面端；开发需要 Node.js、Rust stable 和相应的 Tauri 平台依赖。飞书/Lark 集成额外需要 Go 工具链。

安装依赖并启动桌面开发环境：

```bash
npm install
npm run desktop:dev
```

## 提交 Issue

请尽量提供以下信息：

- 复现步骤、预期行为和实际行为；
- 操作系统、应用版本，以及必要时的 Node.js / Rust 版本；
- 已脱敏的错误信息、日志或截图；
- 最小复现仓库或示例（如适用）。

安全漏洞请不要公开提交 Issue；在项目建立正式安全报告渠道前，请联系维护者私下披露。

## 提交 Pull Request

1. 从最新默认分支创建一个聚焦的分支。
2. 遵循现有代码风格：前端使用 TypeScript + React，Rust 代码通过 `rustfmt` 格式化。
3. 为新增或调整的函数、类和核心变量补充清晰注释；复杂分支、循环、文件系统或 Tauri 调用请添加行内说明。未完成的工作请标记 `todo`。
4. 为关键逻辑补充或更新测试；提交前运行与改动相关的验证。
5. 在 PR 描述中说明改动内容、验证结果、关联 Issue，以及对本地文件、索引、模型请求、日志或写入确认流程的影响。
6. 可见 UI 改动请附截图或录屏，并确认其中不含敏感信息。

建议在提交前执行：

```bash
# 前端类型检查和生产构建
npm run build

# Rust 单元测试；修改 src-tauri/ 时必须执行
npm run rust:test
```

如涉及飞书/Lark 网关，也请构建相应 sidecar：

```bash
npm run sidecar:feishu:build
```

## 设计与安全原则

- 笔记文件属于用户；不得绕过授权范围读取、写入或上传本地文件。
- Agent 发起的 Markdown 修改必须保持“生成 diff → 用户确认 → 校验后写入”的流程。
- 模型密钥和 IM 凭证必须使用系统安全存储；不得写入仓库、测试夹具、日志或截图。
- 日志应可观测、分级可控并进行敏感信息脱敏；避免记录笔记正文、绝对路径和凭证。
- 修改存储结构、索引、导入导出或删除行为时，请优先考虑兼容性、失败恢复和用户数据安全。
